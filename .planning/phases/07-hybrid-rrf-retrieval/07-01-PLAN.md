Use a mock/stub VecStore that returns predetermined (memory_id, distance) tuples. FTS is testable directly via the existing `base` fixture which sets up a real SQLite database with FTS5.

    Then implement `hybrid_search()` as a method on RetrievalEngine:

    ```python
    def hybrid_search(
        self,
        query: str,
        query_embedding: bytes | None = None,
        k: int = 20,
        rrf_k: int = 60,
        vec_store: "VecStore | None" = None,
    ) -> list[tuple[str, float]]:
    ```

    The method:
    1. Runs FTS leg: `Memory.search_fts(query, limit=k)` -- results already ordered by BM25 rank
    2. Runs vector leg (if vec_store, vec_store.available, and query_embedding are all truthy): `vec_store.search_vector(query_embedding, k=k)`
    3. Builds rank dicts: `{memory_id: 1-based-position}` for each leg
    4. Computes RRF: for each unique memory_id across both legs, `score = sum(1/(rrf_k + rank) for each leg the memory appears in)`
    5. Returns `list[tuple[str, float]]` sorted by score descending, limited to top `k` results

    Important details:
    - FTS5 rank values are negative floats (lower = better) -- but we only care about rank POSITION (1st, 2nd, etc.), not the raw BM25 score. enumerate(results, start=1) gives the rank.
    - Vec results are (memory_id, distance) ordered by distance ascending -- again, we use position not raw distance.
    - If only one leg returns results, the memory only gets one RRF term (not penalized for missing from the other leg).
    - Do NOT import VecStore at module top level -- accept it as a parameter to avoid circular imports and keep testability.

    Add the import for `from __future__ import annotations` at the top of retrieval.py if not already present (for the type hint string).
