# Concerns & Technical Debt

_Last updated: 2026-05-06 (T1/T2/T3 wave pass)_

## Known Issues

- **Stale embeddings after content update (correctness bug)**: `Memory.save()` in `core/models.py` recomputes `content_hash` when content changes but does not trigger re-embedding. When reconsolidation refines a memory (appending to `content`) or contradiction resolution rewrites it, the vector in `vec_memories` reflects the old content. KNN queries rank that memory by its stale embedding position, producing incorrect hybrid RRF results. Every `_crystallize_group` in `core/crystallizer.py` and every "refined" action in `core/reconsolidation.py` leaves a stale embedding behind. Source: `core/models.py:184-197`, `core/reconsolidation.py:150-153`, `core/crystallizer.py:359-374`.

- **No busy timeout on apsw connections**: `VecStore._connect()` in `core/vec.py:54-59` opens `apsw.Connection(self._db_path)` without calling `conn.setbusytimeout()`. The peewee connection sets `busy_timeout: 5000` via pragmas (`core/database.py:83`), but apsw connections get the SQLite default of 0ms. If peewee holds a write lock during `db.atomic()` in `Memory.save()`, a concurrent `store_embedding` call raises `apsw.BusyError` immediately. Source: ecosystem-pitfalls research, `core/vec.py:54-59`.

- **No embedding dimension validation**: `VecStore.store_embedding()` and `search_vector()` in `core/vec.py` accept arbitrary `bytes` with no length check. The vec0 table is declared `float[512]` (2048 bytes). If `DEFAULT_DIMENSIONS` in `core/embeddings.py:20` ever changes, or a differently-sized embedding blob is inserted, sqlite-vec returns garbage distances silently — no exception raised. Source: `core/vec.py:65-76`, `core/embeddings.py:20`, ecosystem-stack research.

- **`consolidation_log` migration not atomic**: `_run_migrations()` in `core/database.py:211-244` rebuilds the `consolidation_log` table by executing `DROP TABLE`, `CREATE TABLE`, then a Python loop of `INSERT` statements — all outside any transaction wrapper. A process kill between `DROP` and loop completion permanently destroys the consolidation log with no recovery path. Source: `core/database.py:222-244`.

- **Naive vs UTC datetime mixing**: The codebase uses `datetime.now().isoformat()` (naive, no timezone) in 30+ locations across `core/`, but `core/retrieval.py:382` uses `datetime.now(timezone.utc).isoformat()` (UTC-aware, produces `+00:00` suffix). Mixed formats break `datetime.fromisoformat()` parsing and lexicographic ordering in provenance batch logic (`_compute_provenance_batch`). The relevance engine's `_days_since_last_activity` compares naive `now` against stored timestamps that may be either format. Source: grep across `core/` confirms 30+ naive calls vs 1 UTC-aware call.

- **WAL checkpoint failure silently swallowed**: `close_db()` in `core/database.py:119-128` executes `PRAGMA wal_checkpoint(TRUNCATE)` inside a bare `except Exception: pass`. If the checkpoint fails (e.g., a reader holds an open transaction), the WAL grows unbounded with no warning logged. The checkpoint returns `(busy, log, checkpointed)` columns that are never inspected. Source: `core/database.py:121-124`.

- **Double-logging injections for tier1+tier2 overlap**: `inject_for_session()` in `core/retrieval.py:97-98` calls `_record_injection` for every memory in `tier1 + tier2` without deduplication. An instinctive memory that also scores high in hybrid search appears in both lists. Its `injection_count` is incremented twice, distorting SM-2 scheduling and Thompson sampling parameters. Source: `core/retrieval.py:88-98`.

- **`_session_usage` private attribute accessed externally**: `hooks/pre_compact.py:231` accesses `feedback._session_usage` directly to count used memories for the summary string. This is fragile coupling to `FeedbackLoop`'s internal state. Source: `hooks/pre_compact.py:231`, `core/feedback.py:63`.

## New Issues (T1/T2/T3 waves)

- **`ISSUE_SYNTHESIS_PROMPT` is a single large string — any edit risks silent behavior shifts**: The full synthesis prompt is defined as a single module-level string literal in `core/issue_cards.py:44-205`. It embeds JSON schema, QUALITY RULES 1-10, FRICTION DISCIPLINE, three DECISION-KIND RULES (D1-D3), and a fully-populated example card. Because all sections are concatenated in one f-string, editing one rule can accidentally break indentation of an adjacent rule's JSON template, shift a rule number, or silently remove a closing bracket. There are no unit tests that parse the prompt itself or assert rule count. Any agent or developer editing this prompt must validate the output JSON schema by running test_issue_cards.py *and* a live LLM call. Source: `core/issue_cards.py:44-205`.

- **Importance flow through consolidator is write-site disciplined but easy to regress**: After the T3 wave, `consolidator.py:524-538` trusts the LLM's `card.importance` directly (after clamping) instead of recomputing from a formula. The comment at line 524 is explicit: "trust LLM card.importance (already incorporates somatic prior from Stage 1)". This is correct per design decision D4=C. However, nothing enforces this at the type level — if any future branch inside `_execute_decision()` recomputes importance from its own formula (e.g., a fallback that forgets to use `card_importance`), the card's calibrated importance is silently discarded. The fallback path at line 538 (`min(0.5 + importance_boost, 1.0)`) is intentional for non-card observations but must not be applied to card observations. Source: `core/consolidator.py:524-538`.

- **Kensinger +0.05 friction bump is single-site — regression risk**: The Kensinger affect-encoding bump (`base_importance = min(1.0, base_importance + 0.05)`) is applied exactly once, at `core/consolidator.py:534-535`, guarded by `if card_fields.get("affect_valence") == "friction"`. A comment at line 526 marks this as the "sole site per D1=C". Historically the bump was applied in two places and the duplication caused +0.10 inflation. The single-site invariant is enforced only by comment. Any refactor that moves importance computation into a helper or adds a secondary keep-path must not reintroduce the bump. Source: `core/consolidator.py:526-535`. The ISSUE_SYNTHESIS_PROMPT also documents this explicitly at line 103 ("Kensinger +0.05 friction bump is applied at persistence in consolidator.py, not here — do not pre-apply it").

- **`criterion_weights` and `rejected_options` go through JSON round-trip with no validation**: `extract_card_memory_fields()` in `core/issue_cards.py:396-397` returns `card.get("criterion_weights")` and `card.get("rejected_options")` as raw Python objects (dict and list). The consolidator at `core/consolidator.py:558-559` serializes them with `json.dumps()` before writing to `Memory.criterion_weights` (TextField) and `Memory.rejected_options` (TextField). Any consumer that calls `json.loads(memory.criterion_weights)` gets the dict back — but if the LLM emits a string value for `criterion_weights` (e.g., `"none"` instead of `{}`), `json.dumps("none")` stores `'"none"'` and `json.loads` returns a string, not a dict, silently. There is no schema validation between `extract_card_memory_fields()` and the DB write. Source: `core/issue_cards.py:391-398`, `core/consolidator.py:557-559`, `core/models.py:125-126`.

- **`evidence_obs_indices` validation + demotion-to-orphan logic is fragile to indexing scheme changes**: `_card_evidence_indices_valid()` in `core/card_validators.py:77-89` validates indices against `window_count` (the number of observation windows passed to synthesis). The function is called in `core/issue_cards.py:310-314`: cards whose ALL indices are out-of-range are demoted to orphans. This logic assumes the LLM's `evidence_obs_indices` values are 0-based indices into the `observations[]` array. If the ingestion pipeline ever switches from window-based to turn-based indexing (or changes the observation serialization order), `window_count` would refer to the wrong unit and valid cards would be incorrectly demoted. The linkage between `n_obs` (computed at `core/transcript_ingest.py:857`) and the indices the LLM generates is implicit and undocumented. Source: `core/card_validators.py:77-89`, `core/issue_cards.py:310-314`, `core/transcript_ingest.py:857`.

- **`Memory.affect_valence` is set at write time but retrieval boost is partial (Wave D deferred)**: `affect_valence` is written correctly for card-derived memories (`core/consolidator.py:555`). The retrieval engine applies a `AFFECT_FRICTION_BOOST = 0.02` boost to `friction` memories at `core/retrieval.py:955-956`. However, the boost is only applied in the `_crystallized_hybrid` scoring path, not in the instinctive-tier injection path or in any relevance scoring. The column is read correctly where it is used, but the scope is narrower than the design intends — `neutral` and `delight` valences are stored but never act on retrieval. This is a known deferral (Wave D), not a bug, but it means any test asserting that `delight` memories retrieve differently will fail until Wave D lands. Source: `core/retrieval.py:42,955-961`, `core/consolidator.py:555`.

- **`test_hooks.py` failure was pre-existing at last audit; current run shows 36 passed**: As of 2026-05-06, `tests/test_hooks.py` passes cleanly (36 tests). The prior concern about a pre-existing failure appears to have been resolved during the T1/T2/T3 waves. No further investigation needed unless failures recur. Source: manual run 2026-05-06.

- **Plugin install copies entire project root to cache — dev edits do not propagate without reinstall**: The plugin cache at `~/.claude/plugins/cache/memesis-local/memesis/0.2.0/` contains a full snapshot of the project root (including `core/`, `hooks/`, `scripts/`, `tests/`, `eval/`). Edits made in the working directory at `/Users/emmahyde/projects/memesis/` are NOT reflected in the plugin's active runtime copy until the plugin is reinstalled (a new snapshot is written to cache). This means: (1) hook code changes tested locally pass but the deployed plugin runs stale code, (2) a `core/` module fix is invisible to Claude Code's hook invocations until reinstall, and (3) tests running against the local tree validate different code than what hooks execute. There is no hot-reload or symlink mechanism. Source: confirmed by directory listing of `~/.claude/plugins/cache/memesis-local/memesis/0.2.0/`.

## Security Considerations

- **AWS credentials defaulting**: `core/embeddings.py:37-38` defaults `AWS_PROFILE` to `"bedrock-users"` and `AWS_REGION` to `"us-west-2"` via `os.environ.get()`. The `hooks/consolidate_cron.py` sets `CLAUDE_CODE_USE_BEDROCK` as a process-global default. A misconfigured AWS profile silently uses unexpected credentials.

- **Content hash uses MD5**: `Memory.compute_hash()` in `core/models.py:178-180` uses MD5 for deduplication. MD5 is not collision-resistant. A crafted input could produce a collision that falsely triggers duplicate rejection, preventing memory creation. SHA-256 is negligibly more expensive.

- **FTS5 operator injection mitigated**: `Memory.sanitize_fts_term()` (`core/models.py:117-126`) wraps terms in double-quotes to neutralize FTS5 operators. `tokenize_fts_query()` (`core/models.py:129-160`) applies this to all query terms. `hooks/user_prompt_inject.py` and `core/relevance.py` both use `sanitize_fts_term()`. This concern has been addressed and is preserved here for audit trail.

- **`HOME` mutation in test fixtures**: If `tests/conftest.py` mutates `os.environ['HOME']` for test isolation, this is process-global and unsafe under parallel test execution (`pytest-xdist`). Any test reading `Path.home()` during the mutation window gets the wrong path.

## Performance

- **N+1 `get_embedding()` calls in reconsolidation**: `_rank_by_similarity()` in `core/reconsolidation.py:403-424` calls `vec_store.get_embedding()` individually for the source and each candidate. Each call opens a new apsw connection, loads the sqlite-vec extension, executes the query, and closes. For 3 candidates this is 4 connection cycles. The same pattern appears in `_centroid_similarities()` in `core/graph.py:158-195`. Source: `core/reconsolidation.py:403-424`, `core/graph.py:158-195`.

- **`compute_relevance()` per-memory queries for integration factor**: When `integration_factor` flag is enabled, `RelevanceEngine.compute_relevance()` in `core/relevance.py:122-138` calls up to 4 separate `.exists()` queries per memory (thread membership, tag overlap, causal edges, contradiction edges). For `score_all()` processing every active memory, this is 4N queries. The `_has_tag_overlap()` method (`core/relevance.py:168-202`) iterates each non-meta tag and runs a separate query for each. Source: `core/relevance.py:122-138,168-202`.

- **Consolidation `_build_manifest_summary()` loads all memories**: `Consolidator._build_manifest_summary()` in `core/consolidator.py:167-186` calls `Memory.by_stage()` for each of three stages, loading all non-ephemeral memories into Python to build a text summary for the consolidation prompt. With a growing memory store this becomes increasingly expensive for every consolidation run.

- **Tag co-occurrence `compute_edges()` quadratic check**: `core/graph.py:76-91` iterates all `combinations(mids, 2)` for each tag, and for each pair runs an `.exists()` query to check for duplicate edges. For a tag shared by 20 memories, this is C(20,2) = 190 existence checks, each a separate SQL query. Source: `core/graph.py:76-91`.

- **Token budget approximation**: `CONTEXT_WINDOW_CHARS = 200_000 * 4` in `core/retrieval.py:23` assumes 4 chars/token. The approximation is adequate for English prose but may over-/under-estimate for non-ASCII content or structured data.

## Fragile Areas

- **FTS5 external content sync on bulk operations**: `Memory.save()` overrides keep FTS in sync via `_fts_delete_from_db()` + `_fts_insert()` inside `db.atomic()`. But `Memory.insert_many()`, `bulk_create()`, or raw `INSERT INTO memories` SQL bypass `save()` entirely, leaving `memories_fts` desynced. Currently no code path uses bulk inserts, but adding one without knowing about this constraint would silently break full-text search. Recovery: `INSERT INTO memories_fts(memories_fts) VALUES('rebuild')`. Source: `core/models.py:184-197`.

- **FTS desync on SIGKILL**: The FTS sync sequence in `Memory.save()` is wrapped in `db.atomic()`, which protects against normal exceptions. But `SIGKILL` or OOM between the DELETE and INSERT steps leaves the FTS index missing a row. No startup integrity check or `rebuild_fts()` utility exists. Source: `core/models.py:192-196`.

- **Crystallizer/rehydration tension**: The crystallizer archives source memories after synthesis (`core/crystallizer.py:359-374`). The relevance engine immediately considers rehydrating them based on relevance score (`core/relevance.py:297-358`). Net effect is additive (crystal + potentially rehydrated sources), not the intended compression. Over time this inflates the active memory store. Source: `consolidated/native/crystallization_rehydration_tension.md`.

- **Lock file coordination between hooks**: `hooks/pre_compact.py` and `hooks/consolidate_cron.py` both use `fcntl.flock()` on `ephemeral/.lock`. Both write a snapshot to `.processing-{ephemeral_path.name}`. If both run simultaneously against the same buffer, they overwrite each other's snapshot. The lock protects the read+clear of the original buffer but not the snapshot file. Source: `hooks/pre_compact.py:93-103`.

- **NLTK network download on first call**: `core/relevance.py` and `core/feedback.py` guard `nltk.data.find('corpora/stopwords')` with `try/except LookupError`, falling back to `nltk.download('stopwords', quiet=True)`. In airgapped environments the download fails silently (returns `False`), and keyword extraction degrades to no stopword filtering. Additionally, `nltk.download()` raises `OSError` if `~/nltk_data/` is read-only (containers) — not caught by the `LookupError` handler. Source: `core/feedback.py:24-32`, `core/relevance.py:370-376`.

- **Thompson sampling non-determinism in tests**: `_thompson_rerank()` in `core/retrieval.py:693-714` uses `random.betavariate` without seeding. Any test asserting retrieval order will produce non-deterministic results. Source: `core/retrieval.py:705-711`.

- **Feature flag cache never invalidated during a session**: `core/flags.py:44-68` caches flags on first `get_flag()` call for the lifetime of the process. If `flags.json` is modified while a hook is running (e.g., during a long pre_compact run), the change is invisible until the next process invocation. `reload()` exists but is never called except explicitly. Source: `core/flags.py:44-74`.

- **`_increment_consolidation_count()` not atomic**: `hooks/pre_compact.py:48-53` reads a JSON counter file, increments in Python, and writes back. A read-modify-write race is possible under parallel test workers or unusual hook invocation. This counter controls self-reflection frequency. Source: `hooks/pre_compact.py:48-53`.

- **`ISSUE_SYNTHESIS_PROMPT` structural integrity has no automated gate**: The prompt contains exactly 10 QUALITY RULES and 3 DECISION-KIND RULES, with embedded JSON templates that use doubled braces (`{{`) to escape f-string interpolation. If an edit accidentally un-doubles a brace, the `format(**kwargs)` call at prompt instantiation time raises `KeyError` at runtime (not import time), failing the entire synthesis pass silently under broad exception handlers. No test validates the prompt renders without error. Source: `core/issue_cards.py:44-205`.

- **`synthesize_issue_cards()` demotion path depends on `n_obs` accuracy**: At `core/issue_cards.py:310-314`, `n_obs` is passed as `window_count` to `_card_evidence_indices_valid()`. If the caller (`core/transcript_ingest.py:857`) passes a stale or incorrect `n_obs` (e.g., the observation list was filtered before calling but `n_obs` was captured before filtering), the index range check silently accepts out-of-range indices or incorrectly rejects valid ones. The variable name is `n_obs` but the semantic unit is "number of observation windows in this synthesis batch" — any refactor of the observation batching logic must update this value in lockstep. Source: `core/transcript_ingest.py:857`, `core/issue_cards.py:307-316`.
